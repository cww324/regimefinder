## 1. Current State (Phase 2 — post-ML pivot)

### What We Have
- **365d dataset**: BTC-USD + ETH-USD 5m candles (Feb 2025 – Feb 2026), Postgres
- **1 confirmed signal family**: CA (cross-asset ETH/BTC slope flip), 25 PASS hypotheses (H59–H85, H99)
  - All 25 are variations of the same core mechanism: ETH/BTC 1h EMA20 slope change predicts BTC direction for 30–40 min
  - Effectively 1 independent edge replicated 25 ways for robustness confirmation
- **HMM regime labels**: `rc.regime_labels` — RANGING 41%, TRENDING 25%, VOLATILE 34%
- **Derivatives data in DB**: `rc.funding_rates` (populated ✓), `rc.open_interest` (status unknown), `rc.liquidations` (status unknown)
- **ML pipeline built**: HMM regime discovery + RF hypothesis generator (both operational)

### Key Lessons from Phase 2 (H124–H139)
1. **RF → rule extraction failed**: Single-feature SHAP rules (H125-H133) all FAIL. Multi-feature pair rules (H134-H139) find real gross signal (P>0=0.996-0.998) but hit a ~5-6bps ceiling that can't clear the 8bps cost gate.
2. **The 5-6bps ceiling is structural**: Every pair rule at p70/p30 thresholds converges to ~5-6bps regardless of which features. This is the true signal floor for this class of rules on this data — untradeable.
3. **Data mining vs. theory-first**: H124-H139 were generated by looking at what the RF found, then validated on the same 365d dataset. This is not independent validation. The 25 CA passes came from theory-first hypotheses.
4. **One real signal, not 25**: All confirmed signals are CA-family. No FR, VS, MR, MS, ST, or CD signals confirmed yet.
5. **RF as validator, not generator**: RF is useful as a post-hoc check on theory-driven hypotheses, not as the primary discovery tool on the same data used for validation.

## 3. Hypothesis Backlog — Anchor Candidates

**Strategy**: Anchor-first. Test one clean hypothesis per distinct mechanism. If it passes,
expand (different thresholds, horizons, asset variants, regime gates) to build a family.
Do NOT test all candidates at once — that is data mining.

**Priority order**: Start with mechanisms where data is already available (Phases 1-2), then
build data infrastructure for the rest (Phases 3-4).

## 4. Research Process (Anchor-First)

For each anchor candidate above:

1. **Pre-commit the rule** — write the exact hypothesis (features, threshold, direction, horizon)
   in `hypotheses.yaml` BEFORE looking at any results
2. **Run through pipeline** — WF+bootstrap, gross + bps8 gates
3. **If PASS**: immediately run 2-3 robustness checks (different lookback, odd/even subsample,
   next-bar lag). If those hold, you have an anchor. Then expand the family.
4. **If FAIL**: note why in decision_notes. Move to next anchor.
   - If close (gross passes, bps8 fails): try 1-2 threshold variations. If still fails, move on.
   - Do NOT keep iterating — that is mining.
5. **Anchor expansion** (only after anchor passes + robustness checks):
   - Different horizon (6, 12, 24 bars)
   - Other asset (ETH variant)
   - Regime-gated version (TRENDING/RANGING/VOLATILE)
   - Session-gated version (08-16 UTC)
   - Combined with CA signal

## 6. Things NOT to Do (Guardrails)

- Do not generate hypotheses from RF on the same data used for validation
- Do not test all anchor candidates at once — pick one mechanism, exhaust it, move on
- Do not add more CA family variations — that family is well-explored
- Do not tune thresholds after seeing results — pre-commit first
- Do not interpret P>0=0.99 as "almost passes" — if bps8 fails, it fails
- Do not deploy RF model output directly as a live trading signal
